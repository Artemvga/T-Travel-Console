1. СУТЬ ПРОЕКТА

Проект "Т-Путешествия" — это сервис поиска маршрутов по городам России.
Он вырос из консольной версии и сейчас разворачивается как полноценное веб-приложение:

- backend: Django + Django REST Framework;
- frontend: React + Vite;
- данные: локальный датасет городов, перевозчиков и билетов;
- карта: Yandex Maps API.

Пользователь может:

- искать город и смотреть его транспортный профиль;
- строить маршрут между городами;
- задавать дату и время отправления;
- выбирать приоритет маршрута: быстрее, дешевле, оптимально;
- ограничивать поиск по видам транспорта;
- задавать обязательный транзитный город;
- строить маршрут без пересадок или с пересадками;
- смотреть маршрут на карте;
- сохранять маршруты в избранное после авторизации.


2. ОСНОВНАЯ ИДЕЯ АРХИТЕКТУРЫ

Проект состоит из 4 крупных частей:

1. Источники данных в data/
2. Скрипты подготовки и генерации данных в scripts/
3. Backend API в backend/
4. Frontend интерфейс в frontend/

Сейчас проект работает так:

- города импортируются из data/cities/cities.json;
- перевозчики импортируются из data/planes, data/trains, data/buses/by_operator, data/commuter_trains;
- билеты импортируются из JSON или из временных JSONL-файлов генератора;
- backend строит маршруты не через полный граф в памяти, а через поэтапный поиск по билетам в БД;
- frontend запрашивает backend API и показывает страницы города, маршрута, карты, аккаунта и избранного.

5. КАК РАБОТАЕТ ПОСТРОЕНИЕ МАРШРУТА

Маршрут строится не через наивный полный DFS по заранее загруженному графу в памяти, а через поиск по билетам в базе.

Текущее ядро поиска находится в:

backend/apps/routes/services.py

Логика поиска:

1. Находим город отправления, город прибытия и при необходимости транзитный город.
2. Формируем временную точку отправления.
3. Идем по состояниям поиска:
   - текущий город;
   - текущее доступное время;
   - уже пройденные сегменты;
   - набор уже посещенных городов;
   - суммарная цена;
   - суммарное расстояние.
4. Для каждого состояния выбираем ограниченный набор подходящих билетов.
5. Не даем маршруту возвращаться в уже пройденные города.
6. Учитываем буфер на пересадку.
7. Сортируем готовые пути по приоритету пользователя.

Поддерживаемые режимы:

- прямой маршрут;
- маршрут с пересадками;
- маршрут через обязательный транзитный город;
- только автобус;
- только поезд;
- только электричка;
- смешанный маршрут;
- максимум до 5 пересадок.

Для frontend маршрут отдается уже в сериализованной форме:

- waypoints
- path_coordinates
- segments
- total_price
- total_duration_minutes
- total_distance_km
- transfers_count
- transport_legend
- map_mode
- map_enabled


6. КАК РАБОТАЮТ КАРТЫ

Маршрутная карта реализована на Yandex Maps API.

Текущий подход гибридный:

- самолет:
  рисуется дугой;

- автобус:
  строится через road routing Yandex;
  если road routing не удался — рисуется fallback-кривой;

- поезд:
  пробует transit/multiroute сценарий;
  при неудаче — fallback-линия;

- электричка:
  аналогично поездной логике, но своим цветом.

Каждому виду транспорта соответствует свой цвет.
Также у маршрута есть легенда, где показано суммарное время по видам транспорта.

На странице города используется отдельная карта города:

- центрируется на выбранном городе;
- ставит маркер;
- показывает радиусную зону;
- работает внутри единого контейнера с городской информацией.

8. ЧТО ЕЩЕ НЕ ДОВЕДЕНО ДО КОНЦА

Есть уже реализованный каркас под production-scale, но некоторые большие шаги еще не завершены полностью:

- PostgreSQL-конфиг реализован, но конкретный production-инстанс нужно поднимать отдельно;
- генератор умеет работать по новой логике, но полный массовый реимпорт на 15 млн билетов не запускается автоматически при каждом dev-старте;
- часть текущих data/*.json может быть пустой после незавершенной генерации, поэтому dev-импорт сейчас умеет безопасно fallback-ить на JSONL;
- для очень больших объемов данных желательно окончательно перевести все тяжелые витрины на PostgreSQL и snapshot/summary-слой.


9. КАК РАБОТАЕТ АВТОРИЗАЦИЯ

Авторизация реализована через Django Token Auth.

После входа:

- frontend сохраняет токен в localStorage;
- каждый API-запрос автоматически подставляет `Authorization: Token ...`;
- backend возвращает профиль пользователя и историю;
- пользователь получает доступ к избранному.

Авторизованный пользователь может:

- сохранить найденный маршрут в избранное;
- открыть страницу избранного;
- видеть количество сохраненных маршрутов;
- видеть последние поиски.


10. КАК РАБОТАЕТ ИЗБРАННОЕ

Избранное реализовано через модель RouteFavorite.

Хранится:

- пользователь;
- from_city;
- via_city;
- to_city;
- departure_at;
- priority_mode;
- route_title;
- route_signature;
- route_data;

Особенности:

- сохранить маршрут можно только авторизованным пользователем;
- используется сигнатура маршрута по сегментам;
- защита от дублей идет по:
  user + route_signature + departure_at

16. ЛОГИКА ГЕНЕРАЦИИ БИЛЕТОВ

Генератор билетов учитывает:

- профиль города;
- тип транспорта;
- расстояние между городами;
- допустимую длину маршрута;
- forced corridors;
- роль хабов.

Plane:
- коэффициент пути 1.02–1.10;
- скорость 680–820 км/ч;
- короткое время по сравнению с поездом.

Train:
- коэффициент пути 1.08–1.35;
- скорость 45–95 км/ч;
- ограничение на заведомо нереалистичные длительности.

Bus:
- коэффициент пути 1.10–1.45;
- скорость 35–70 км/ч;
- нельзя делать прямые сверхдальние маршруты как норму.

Electric train:
- только короткие региональные плечи;
- максимум около 250 км;
- строится только между городами с commuter-профилем.

- маршрутами,
- городами,
- картой,
- поиском,
- аккаунтом,
- избранным,
- генератором данных,
- импортом в backend.

То есть ядро приложения уже собрано.

Главный следующий тяжелый этап — не интерфейс, а инфраструктура и масштаб:

- окончательно перевести рабочий контур на PostgreSQL;
- полностью пересобрать билетную базу большого размера;
- проверить стабильность поиска и витрин на больших объемах данных.


19. КОРОТКОЕ РЕЗЮМЕ

Т-Путешествия — это веб-сервис поиска маршрутов по России.

Что есть уже сейчас:

- backend API;
- frontend интерфейс;
- города;
- маршруты;
- карта;
- избранное;
- авторизация;
- генератор билетов;
- импорт данных;
- транспортные профили городов.

Что является основой проекта:

- data/ — данные;
- scripts/ — подготовка и генерация;
- backend/ — логика и API;
- frontend/ — интерфейс.
